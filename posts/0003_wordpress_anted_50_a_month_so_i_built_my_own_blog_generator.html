WordPress Wanted $50 A Month So I Built My Own Blog Generator

September 13th, 2025

TL;DR

I built a minimal, static, SPA blog generator and CLI tool named Quill. You can find it on GitHub here and it’s actively being worked on.

$50 a month for a syntax highlighting plugin!? I took that as both an insult and a challenge.

So I left WordPress behind and built my own static blog generator. If you’re reading this, it means it worked.

Before I started, I explored other site generators. I wanted something:

Minimal - I don’t need a fancy UI. I prefer Markdown and a terminal.

SPA-based - client-side routing just feels smoother these days.

Low-level - I still want to drop in a few lines of CSS when I need to.

Nothing quite fit. Most were bloated, poorly documented, or annoying to install. So I did what any mildly insulted developer would do - I wrote my own and called it Quill.

Design Philosophy

I’m making a habit of setting the ‘commandments’ of what I’m building upfront. It’s surprising how many decisions that can make for you in the designing and building process.

For Quill I declared:

Don’t Scale

No bells, no whistles. Minimal dependencies. Generate small, focused blogs that do one thing well.

Quill doesn’t even support pagination, and I’m not sure I’ll add it.

Sensible defaults, flexible when needed

Everything is configurable, but nothing has to be. Start simple, customize only if you want.

I believe I’ve borrowed this from the Ruby-on-Rails doctrine. It’s fun to twiddle the knobs, but the focus should be on writing and publishing.

Posts aren’t programs

Content should stay close to raw text. The tooling gets out of the way so you can focus on words, not code.

I started with having each blog post written as HTML but quickly found it got in the way of just writing. I was able to have Quill support embedded HTML in the Markdown files that create the blog posts but found the simpler format to be superior.

Just write

The workflow should encourage writing, not configuring. Remove friction, remove distractions.

Following on from the previous point - I want to write and publish, not fiddle with code styling plugins.

How it works

Thankfully the core of static site generation is straight forward: take some input, produce some HTML, CSS, and JavaScript. Once you’re happy, host it somewhere.

The input for Quill is Markdown files. When writing a blog post I rarely feel the need for fancy formatting and Markdown covers all the bases. Turning Markdown into HTML is already a solved problem so I decided to sink some time into writing a bespoken SPA router (surprisingly, it only took 50 lines of code!) and setting some sensible CSS defaults.

If you download Quill and run

quill init

You’ll see a posts/ directory with a 0001_hello_world.md file inside (the 0001 is used for indexing - posts are rendered on the homepage by their index in descending order).

If you then start up your Quill server:

quill server

You’ll be able to see your rendered blog, make changes to your 0001_hello_world.md, and see changes in real-time.

Give that ‘Hello, World!’ link a click and see the SPA navigation in action.

The CSS is intentionally minimal but fully customizable - you, the author, have full ownership and access to the styles.css so can tweak until you’re satisfied.

Hopefully this demonstrates the the eagerness for Quill to get the blog, and writing content, up and running as quickly as possible.

Once you’re done adding blog posts, you’ll need to host your blog somewhere.

I spent an afternoon writing some intricate scripts to push blogs to AWS’s S3 and enable static sites, but couldn’t shake the feeling of ‘this feels like it should be easier’… and the little voice was right. Github Pages are the answer.

As simple as running quill build, pushing the build to your GitHub IO page and you’re done.

What’s next?

I’ll keep expanding Quill with more CLI options and templates - but always with the same goal: make writing the focus

If you’re tired of fighting with WordPress, looking for something small with full control, or just want to write Markdown, give Quill a try.

This project began out of frustration but it turned into something satisfying. Building Quill reminded me that we still can build our own tools - simple, fast, and ours.
